## Active Decisions

- Market history stored as separate `market_history.json` (not in `market_proxies.json`) to avoid breaking existing consumers
- Generation mix uses same `EntsoeGenerationCollector` class with expanded params, separate from French nuclear `generation_forecast.json`
- Time resolution: store each source at native resolution, defer alignment to consumers — see [ADR-001](../docs/decisions/ADR-001-time-resolution-strategy.md)
- In-repo memory (this file + topic files) over auto-memory, per agent-ready-projects ADR-001
- OpenMeteo `OPENMETEO_SEMAPHORE_CAP` = collector count (not lower) — empirically required to avoid Open-Meteo's CDN per-source cooldown affecting late-scheduled collectors. See `collectors/_openmeteo_shared.py` docstring.
- **Multi-collector concurrent upstream pattern**: when adding a new collector that hits an API already used by other collectors, assume the upstream applies per-source rate-limits, CDN cooldowns, or "no data" 4xx responses that look like transients. Five incidents in `memory/gotcha-log.md` (ENTSO-E 503 cascade, Luchtmeetnet 429, Open-Meteo 429 storm, Open-Meteo CDN cooldown timeout, TenneT 422→429) all share the same shape. Default prescription: (1) shared semaphore cap ≥ collector count, (2) per-location retry with exponential backoff, (3) classify non-retryable HTTP statuses (422/400/401/403/404) at the collector level instead of letting BaseCollector retry them. See `collectors/_openmeteo_shared.py` for the cap pattern and `collectors/_http_classifier.py::raise_if_permanent` for the canonical HTTP-status bail-out (extracted from `tennet.py` on 2026-06-07 specifically so the next ENTSO-E/NED/GIE incident is two lines away from being fixed). Any `_fetch_raw_data` permanent-exit path MUST raise `NonRetryableError`, never plain `Exception`/`ValueError` — the outer retry loop catches anything else.
- **Parallel hard-coded registries**: when you see two-or-more parallel lists keyed on the same identifier (anywhere — Python, YAML, JSON, doesn't matter), consolidate before adding the third instance. Two incidents: `data_quality.py` had 3 lists for missing-dataset severity (collapsed in 39a94d6 to one `DATASET_MISSING_SEVERITY` dict); `collect-data.yml` had 2 lists for publishable feeds (completeness tripwire at line 54 + docs-prepare at line 118) — adding #3's `nordic_hydro` to one but not the other was the BLOCKER flagged by the multi-model review battery on c40a53b. Hard rule: when adding the third item, collapse OR add a test that asserts the lists match.
- **Warn-only bedding-in audits**: when flipping any CI guard from `--warn-only` to fail-mode, audit the actual warnings emitted during the bedding-in window — don't just count green runs. Two incidents: the 2026-06-08 catalog-drift warning was on the record but unread (run was overall green); the `_TS_PATTERN` date-only false-positive class was swallowed for the entire bedding-in period and only surfaced when fail-mode flipped on 2026-06-10. Corollary: introspection tools (shape_signature, drift detectors) have their own assumptions — exercise them against real published payloads, not just synthetic fixtures, before trusting them as gates.
- **Silent quality-gate skip** (3-incident pattern, broadened 2026-06-08): a registered behaviour that's expected to enforce/surface something silently no-ops at a downstream layer, and the absence looks identical to "data is clean". Three incidents now: GoogleWeather `API_KEY_INVALID` returned success exit codes for 7 months while the API was 401-ing every call (validator path); `validate_value_ranges` silently no-op'd on 2-level-nested feeds for ~3 months (validator path); TenneT custom `collect()` override bypassed BaseCollector's metadata auto-inject so `balance_delta_status` + `collector_quality_issues` populated correctly in memory but never reached the published file (publish-path drop, 2026-06-08 dispatched-run discovery — `ab3dcd4`). Hard rule, now generalised: when wiring a new field/check/signal, **trace it end-to-end through every layer until you observe it on disk or in the operator log**. Don't trust that "registered = surfaced". Two-layer corollary: when overriding a BaseCollector method, audit the WHOLE contract (reset → fetch → parse → normalize → validate → metadata → inject → dataset), not just the parts the override touches. Pattern signature: unit tests green, live run silently drops the new field.
- **Derive classifications from history, don't hand-maintain exception lists** (2026-06-14): a hand-curated allowlist of known-bad cases is *reactive* — it only protects against what already broke, never the next instance. The schema-drift `VOLATILE_SHAPE_FEEDS` allowlist grew 1→3 in two days (air_quality_buurt 06-13; cross_border_flows + calendar_features 06-14), each added only *after* it failed a production run. Fix: `scripts/detect_schema_drift.py::derive_volatile_feeds()` derives the volatile set from committed shape history (>1 hash at the same schema_version = data-driven churn), unioned with the declared seed. This got *ahead* of the problem — on first run it proactively flagged `grid_imbalance`/`market_history`/`market_proxies`, three not-yet-broken feeds. General rule: when you catch yourself adding the Nth entry to a hand-maintained exception list for the same recurring class, ask whether the membership can be *computed* from observed history/state instead. Irreducible residual: the very first occurrence of a novel case still can't be auto-distinguished from a real problem (nothing to compare) — but that's once-per-case, not the recurring pain. Sibling to "warn-only bedding-in audits" (both about not trusting a guard's surface-green state).
