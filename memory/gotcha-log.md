### `overall_status=error` on weekends is a market-staleness artifact, not a regression (2026-06-14)
**Problem**: Wrap-up smoke test on Sunday 2026-06-14 found the published quality report at `overall_status=error, total_issues=4`, vs the documented steady state of `warning, total_issues=2`. Looked like a degradation snuck in.
**Root cause**: Two of the four are weekend artifacts: `market_proxies` and `market_history` flagged `staleness` (`Newest data is 56.0h old, threshold 48h`). Carbon (EUA) and gas (TTF) markets don't trade Sat/Sun, so on Sunday the newest data is Friday's close (~56h old). `validate_staleness` defaults to 48h (`utils/data_quality.py:831`); `STALENESS_OVERRIDES` (`:1021`) raises it for slow feeds (`gas_storage`=96, `nordic_hydro`=672) but has no entry for the weekday-only market feeds, and 48h can't span a weekend. The other two issues are the documented `grid_imbalance` soft-gate + `load_forecast` #30 watch item.
**Fix**: None applied (non-blocking — `error` doesn't abort publish, only `critical` does). Diagnosed + documented. Proposed fix: add `market_proxies`/`market_history` to `STALENESS_OVERRIDES` with a weekend-spanning floor (e.g. 96h, matching gas_storage), or weekend-aware staleness. **Filed as #36.**
**Lesson**: The documented "warning/2 steady state" was a *weekday* observation — quality status is day-of-week dependent. Same flavor as the schema-drift volatile-feed class: a threshold (48h staleness) that ignores a feed's legitimate data cadence (markets closed weekends). When a status field is meant to signal real problems, recurring benign-but-noisy trips erode its signal value — the market-staleness `error` every weekend makes a real market-data outage look identical to a normal Sunday.

### Schema-drift false positives kept recurring → made volatility self-classify from history (2026-06-13 / 2026-06-14) [RESOLVED]
**Problem**: Three CI failures in two days, all the same class — a feed's shape hash changed with no `CURRENT_SCHEMA_VERSION` bump, so the fail-mode tripwire aborted publishing — but none were real schema changes. (1) 2026-06-13 `air_quality_buurt`: luchtmeetnet encodes the nearest-online RIVM station set + per-station pollutant set as dict keys; a station dropping offline and returning flipped the hash (`b343fc99`↔`c30a221a`). (2) 2026-06-14 `cross_border_flows`: `data.flows` samples one representative hour and the `NL→GB` interconnector reported no flow that hour, so the border key dropped from the shape. (3) 2026-06-14 `calendar_features`: `metadata.upcoming_holidays` flipped from an empty list (`value_shape: null`) to a list-of-dicts when a holiday entered the lookahead window.
**Root cause**: `shape_signature` treats "which data happened to be present" as part of the schema — it samples one timestamp_map entry, and distinguishes empty-vs-populated collections. For feeds whose key set is data-driven, the fingerprint legitimately churns day-to-day. The first fix (`VOLATILE_SHAPE_FEEDS` allowlist, 06-13) was **reactive**: a feed only stopped paging *after* it broke CI and a human added it. The set grew 1→3 in two days; feed #4 would fail the same way.
**Fix**: Added `derive_volatile_feeds()` (`scripts/detect_schema_drift.py`): walks the last N committed sidecars and marks a feed volatile if it ever showed >1 `shape_hash` AT THE SAME `schema_version` (data-driven churn; versioned migrations move the version, so they're excluded and can't mask a later real break). `main()` unions this with the declared set, so the allowlist becomes a documented seed/override + thin-history fallback rather than a hand-maintained list. `--volatility-window N` (default 60; 0 disables). Validated against real history: the classifier proactively flagged `grid_imbalance`, `market_history`, `market_proxies` — three feeds already volatile (rolling windows / type flips) but not yet broken in CI, i.e. three future false positives defused before paging. A stable feed that then truly breaks is still enforced (exit 1). Also marked all three diagnosed feeds in the declared seed set. 22→28 drift tests.
**Lesson**: An allowlist of known-bad cases is reactive hardening — it protects against what already bit you, never the next one. To get *ahead* of a recurring false-positive class, derive the classification from observed history instead of hand-curating it. The irreducible residual: a feed's very *first* unversioned deviation can't be distinguished from a real break by anything (one prior hash) — but that's a per-feed-once event, not the recurring paging that was the actual pain. Sibling to the `_TS_PATTERN` lesson below (introspection tools have data-shape assumptions) — here the fix was to stop hand-maintaining the exception list and let history maintain it.

### `actions/setup-python@v5` floating major tag still resolves to a Node 20 build (2026-06-14) [RESOLVED]
**Problem**: Migrating actions off Node 20 (forced to Node 24 by GitHub on 2026-06-16), bumped `actions/checkout@v4→v5` and `actions/setup-python@v4→v5`. Tests still emitted the Node 20 deprecation warning — now pointing at `setup-python@v5`. A second round was needed (`→v6`).

SHA-pinning + Dependabot sidesteps the whole ambiguity: you pin a known-good commit and let automation move it forward. A composite action (codecov) can also drag in a deprecated transitive action — check its `action.yml`, not just its own runtime.

Neither `energy_data_fetchers/` nor `weather_data_fetchers/` exists — the real code is in `collectors/` and `utils/`. coverage.py measured only `utils/`, so the `coverage.xml` uploaded to Codecov never reflected `collectors/` at all (pre-existing since the original CI setup).

### Schema-drift tripwire's `_TS_PATTERN` only matched ISO-T, missed `YYYY-MM-DD` date-only keys (2026-06-10) [RESOLVED]
**Problem**: Flipped schema-drift tripwire from `--warn-only` to fail-mode (`8e63148`). First dispatched run after the flip failed at the tripwire step. CI reported within-feed shape drift on `market_history.json` + `market_proxies.json` with no `CURRENT_SCHEMA_VERSION` bump — looked like a silent shape regression. It wasn't. Decrypted both feeds: shape unchanged; only the rolling date window had advanced.
**Root cause**: `utils/shape_signature.py:35` `_TS_PATTERN = re.compile(r'^\d{4}-\d{2}-\d{2}T')` required the `T` separator. Three dict locations across the two feeds (`market_proxies.gas_ttf.history`, `market_history.carbon_eua.data`, `market_history.gas_ttf.data`) keyed rolling history windows by `'YYYY-MM-DD'` (date-only, no `T`) — produced by `market_proxies.py:490` / `:504` via `hist.index[-1].strftime('%Y-%m-%d')`. The collapse predicate `all(_is_timestamp_key(k) for k in data.keys())` returned False, so the signature enumerated every date individually. Each day the window rolled (oldest drops, newest appears) → different signature → different hash. `--warn-only` had been silently swallowing this false-positive class for the entire bedding-in period.
**Fix**: Broadened the regex to `r'^\d{4}-\d{2}-\d{2}([T ].*|$)'` so date-only keys also collapse to a `timestamp_map`. Regenerated `data/_shape_signatures.json` from existing decrypted feed payloads in the SAME commit so the next CI run compared new-logic-vs-new-logic. Only the 2 affected feeds' hashes changed; the other 17 were byte-identical. Added 3 tests: `test_date_only_keys_collapse`, `test_date_only_value_shape_change_still_detected`, `test_market_history_stable_across_days`. Re-dispatched workflow → passed cleanly (run `27279671823`).
**Lesson**: When flipping a CI guard from `--warn-only` to fail-mode, audit what `--warn-only` was actually swallowing during bedding-in — not just count green runs. Two of the three "warned but didn't fail" runs (CI run `27225339687` 2026-06-09 17:54Z + earlier dispatches) were emitting `Within-feed shape drift on 2 feed(s)` warnings that no one read because the run was overall green. Also: **introspection tools have their own assumptions** — `shape_signature` was tested with synthetic ISO-T payloads but never against real `market_proxies.py` output. Run the framework against actual published payloads before trusting it as a CI gate. Where: `utils/shape_signature.py:32-44`. Sibling lesson to the silent-quality-gate-skip pattern in MEMORY.md Active Decisions, but applied to introspection tooling rather than publish-path validation.

### Shared OpenMeteo Semaphore(5) caused buurt collector timeouts (2026-06-07) [RESOLVED]
**Problem**: After PR #18 merged (shared `Semaphore(5)` across 6 OpenMeteo collectors), two consecutive workflow_dispatch runs (27086755763, 27087174593) failed identically: late-scheduled collectors (offshore wind + both buurts) consistently got `Connection timeout to host api.open-meteo.com` while early strategic/demand collectors succeeded. The data-quality gate correctly blocked publish, but air_quality_buurt envelope verification was blocked.
**Root cause**: Open-Meteo's CDN applies a per-source connection-cooldown window after a request burst. With cap=5 < 6 collectors, 28 location requests queued FIFO behind the shared semaphore; the last ~7 (offshore + buurt) arrived during the cooldown. Pre-#11 the 6 collectors had their own `Semaphore(1)` each — 6 parallel sessions instead of one serialised queue — which avoided the trigger.
**Fix**: PR #23 — bumped `OPENMETEO_SEMAPHORE_CAP` from 5 → 6 (== collector count), restoring the effective pre-#11 concurrency pattern. Also added `fetch_location_with_retry` with exponential backoff (1s → 2s → 4s, max 3 attempts) per location for belt-and-suspenders resilience against any future transient.
**Lesson**: When sharing a budget across N independent consumers, FIFO < N means the late arrivals can hit upstream throttling that the early ones triggered. Cap should equal consumer count unless the consumers' arrival pattern is interleaved by design.

### Worktree path mismatch broke 4 of 12 reviewer agents (2026-06-07)
**Problem**: Created git worktrees at `/tmp/wt-prN` via bash for the multi-model review battery on 4 PRs. 4 of 12 spawned reviewer agents (Opus code-reviewer for PRs #16/#18/#19 + Sonnet for #16) returned "I cannot review code I cannot read" — their Read tool couldn't access the paths.
**Root cause**: On Windows, bash's `/tmp` maps to `C:\Users\<user>\AppData\Local\Temp` (a separate per-user temp dir, not the system `/tmp` that POSIX expects). The reviewer agents tried `/tmp/wt-prN` and got "not found"; only the agents that happened to try the Windows path succeeded.

**Positive side-effect**: Three of the four agents that couldn't read the code refused to fabricate findings (cited the "When something is unclear, ask rather than guess" harness-defense rule), which is the desired behavior. The fix is upstream — don't put them in that position.

### Collector's `_fetch_raw_data` raised ValueError when it meant "permanent failure" (2026-06-07) [RESOLVED]
**Problem**: `EntsoeHydroCollector._fetch_raw_data` raised `ValueError("No reservoir data returned")` when all requested zones returned empty after their per-zone retries. Code-reviewer agent flagged this as a BLOCKER: `BaseCollector._retry_with_backoff` catches `except Exception` and triggers up to `max_attempts` more outer retries — each re-running all zones, all returning empty, all raising ValueError again. The bail-out intent was completely defeated.
**Root cause**: There are two distinct exception contracts in `BaseCollector` and they look identical from the inside of a collector:
  - Anything that inherits from `Exception` → caught by the outer retry loop, retried with backoff.
  - `NonRetryableError` (issue #25) → re-raised immediately, no further attempts.

  When a collector decides "I've tried everything I can; the upstream is permanently empty for this window", it must raise `NonRetryableError`, not a plain `ValueError`/`RuntimeError`. The exception name is the signal, not the message.
**Fix**: Changed the raise site to `raise NonRetryableError("No reservoir data returned...")`. Added a regression test that mocks all zones to return None and asserts exactly `num_zones × 1 outer attempt = num_zones` `_retry_single` calls (not `num_zones × max_attempts`). The test would have failed loudly under the buggy ValueError path.
**Lesson**: Any "we tried, it's not coming back" exit from `_fetch_raw_data` is a `NonRetryableError`, never a plain Exception. If a future collector adopts the `_http_classifier.raise_if_permanent` helper, that's already correct; the trap is the *manual* raise sites where it's tempting to reach for `ValueError`. Code comment in `collectors/entsoe_hydro.py` now documents this.

### Three parallel registries with the same key shape — collapse before adding the third (2026-06-07) [RESOLVED]
**Problem**: `utils/data_quality.py` grew three lists for missing-dataset severity: `REQUIRED_DATASETS` (critical), `EXPECTED_DATASETS` (info-only logging), `WARNING_IF_MISSING_DATASETS` (warning, added today for the #25 soft-gate). Each was a `List[str]` keyed on dataset name. The `PipelineQualityReport.status` property and `validate_pipeline` loop consulted all three with implicit precedence. `EXPECTED_DATASETS` was effectively dead — it duplicated the critical entries plus a few info-flake ones, and didn't influence status at all. The refactoring-guide reviewer caught this explicitly: "do this now, before a 4th list".
**Root cause**: Each registry was added in a different commit for a different bug (a74f662, 7673750, etc.). Nobody made the structural call to consolidate because each individual list looked reasonable. The shape "three lists keyed on the same identifier, with implicit severity ranking encoded in which list a name appears in" was only visible from above.
**Fix**: Replaced with a single `DATASET_MISSING_SEVERITY: Dict[str, str]` where values are `'critical' / 'warning' / 'info'`. `validate_pipeline` reduces to one pass; the status ladder consults a single registry. The three named lists are preserved as back-compat shims derived from the dict so any importer keeps working.
**Lesson**: When you reach for a *third* parallel registry with the same key shape (the same identifier in each, the same lookup site iterating all of them), consolidate then. Two parallel lists is the limit before the precedence becomes implicit and ambiguous. The compile-time signal is "I'm about to add another list" — if you're naming it `*_IF_MISSING_*`, that's already a hint that severity is the real dimension and you want a dict keyed by name with severity as value.

### Schema-drift tripwire conflated catalog drift with shape drift (2026-06-08) [RESOLVED]
**Problem**: While verifying the 2026-06-07 daily run as part of the next-session pickup, the 17:07 schedule run's schema-drift step reported `feeds added: ['air_quality_buurt.json']` with `::warning::Schema drift detected ... but schema_version did not change`. Confirmed by diffing the two sidecar commits (`10fe22e` baseline vs `8590b64`): the luchtmeetnet collector failed transiently in the 16:58 dispatch baseline run and recovered by the 17:07 schedule run. The "added" entry was a transient collector recovery, not a schema event. Under the current `--warn-only` mode this was harmless noise, but the 2026-06-21 plan to flip to fail-mode would have errored every CI run that contained even one transiently-flaky collector.
**Root cause**: `scripts/detect_schema_drift.py` treated `feeds_changed` (within-feed shape diff), `feeds_added`, and `feeds_removed` as the same severity class — any of them without a `CURRENT_SCHEMA_VERSION` bump → exit 1. But `schema_version` versions the envelope/migration shape of existing feeds, NOT the catalog of which feeds the pipeline currently publishes. A collector flaking off and back on is operational; a collector being retired is operational; only a within-feed shape change is the silent-shape-break failure mode the tripwire exists to catch.
**Fix**: Split the fail-mode classification in `detect_schema_drift.py:main`. `feeds_changed` without version bump → exit 1 (the real target). `feeds_added` / `feeds_removed` without `feeds_changed` → emit `::warning::Catalog drift: ...` and exit 0, even in fail-mode. Mixed case (within-feed shape change + catalog drift) → fail (shape change dominates). Added three tests: `test_feeds_added_only_exits_0_with_warning`, `test_feeds_removed_only_exits_0_with_warning`, `test_within_feed_change_plus_catalog_drift_still_fails`. The original `test_drift_without_version_bump_exits_1` continues to pass because it uses `feeds_changed`. Module docstring updated to call out "Catalog vs shape drift" explicitly.
**Lesson**: When a tripwire fires three different events through the same exit-code path, ask whether the user (the future operator reading the CI alert) needs the same response for all three. Here the answer was no: shape drift is "did someone forget to bump the version?" while catalog drift is "did a collector come back / get retired?" — two completely different operational questions. The fix preserves the original silent-shape-break detection (which was the tripwire's whole purpose per #27) while removing the false-positive class that would have blocked the 2026-06-21 fail-mode flip. Also a reminder: when scheduling a "flip to fail-mode" date based on N weeks of clean warn-only runs, audit the actual warnings during the bedding-in period — don't just count green runs. The 17:07 warning was on the record; only the cross-session pickup spotted it.

### TenneT `balance-delta` endpoint retired without library update — secondary signal failure killed primary signal (2026-06-08) [RESOLVED]
**Problem**: Issue #25's original diagnosis (422→422→429 cascade on `balance-delta-high-res`) was correct when filed but the API has since changed: TenneT retired the `balance-delta-high-res` endpoint, renamed to `balance-delta`, and the new endpoint now returns HTTP 404 for every window tested (yesterday → 30 days back via `scripts/probe_tennet_windows.py`). `settlement_prices` continues to work normally (96 rows/day = full 15-min resolution). Because `_fetch_raw_data` ran both calls inside one try-block, the dead `balance_delta` endpoint's 404 propagated up and the classifier from 7673750 (correctly) bailed out the entire fetch — taking the working `settlement_prices` payload down with it.   1. **Library lag**: `tenneteu-py` still calls `/publications/v1/balance-delta` but TenneT's actual endpoint either renamed again or is genuinely retired. The library hasn't been updated.
  2. **Coupled error path**: A secondary signal's permanent error killed the primary signal because they shared a try-block. The classifier did its job; the structure around it forced an all-or-nothing outcome.
**Fix**: Split `_fetch_raw_data` so each API call has its own try-block. `settlement_prices` keeps the existing classifier behaviour (permanent → `NonRetryableError`, transient → retry). `balance_delta` permanent errors (404/422/400/401/403) are caught and substituted with `None`; transient errors still propagate so the retry loop has a chance. **Lesson**: When a try-block wraps multiple API calls whose failures have different operational meanings, split it. The classifier-bail-out pattern (`NonRetryableError`) is correct for "no point retrying this", but conflating "primary signal failed" with "secondary signal failed" forces both to take the same action — and the primary's action is "give up entirely" which silently retires the whole feed. The second insight: when an issue's diagnosis ages, re-probe the upstream before re-acting on the original analysis. The 2026-06-08 root-cause work changed the recommended fix from "investigate window boundary" to "split the try-block" because the API shape changed under us in the intervening weeks.

### Custom `collect()` override silently bypassed BaseCollector's metadata + quality-issue contract (2026-06-08) [RESOLVED]
**Problem**: Dispatched the daily collect-data workflow to verify the 4-issue bundle's changes (4c59378) on live CI. The run succeeded — `Catalog drift: 1 added (grid_imbalance.json)` warned correctly, settlement_prices flowed through (96 records), `balance_delta` got HTTP 422 and the classifier correctly bailed out. Unit tests at 599 passing, live behaviour broken at the publish boundary.
**Root cause**: `TennetCollector` has a custom `collect()` override (line 478) that does NOT call `super().collect()`. It calls `_create_dataset(parsed_data, start, end)` (line 538) which builds its own metadata dict from scratch with seven hard-coded fields (`data_type`, `source`, `units`, `country`, `start_time`, `end_time`, `data_points`, `collection_timestamp`, `api_version`). The unit-test-passing-but-live-broken signature is the giveaway. A multi-model review battery looking at the bundle would have caught this if it had asked "where do the new fields actually surface on disk?" — adding that question to the review prompt next time.

### ENTSO-E "no data" upstream gap hard-failed the whole publish (2026-07-01) [RESOLVED, #38]
**Problem**: Daily run 28464598156 exited 1 with `CRITICAL DATA MISSING after 3 retry rounds: ['entsoe', 'entsoe_de']`. Every other feed (20 of them, incl. the alternative NL day-ahead sources EnergyZero/EPEX/Elspot) collected fine, but the run failed and **nothing published** — so Augur got nothing at all, not even the fresh fallback prices. A plain re-run failed identically.
**Root cause**: NOT a code bug — an upstream ENTSO-E Transparency Platform data gap. Live probing (with the real key) showed NL/DE day-ahead prices present up to 2026-06-29 23:45 and absent from 2026-06-30 onward; past windows returned data normally. `entsoe-py`'s `query_day_ahead_prices` fetches `[start-1d, end+1d]`, truncates to the window, and raises `NoMatchingDataError` when nothing remains. The old code treated that generic `NoMatchingDataError` like a crash: 3 fast retries × 3 retry rounds @ 300s (~16 min wasted) → `SystemExit(1)` → publish blocked. Two gates enforced the block: the orchestrator's `SystemExit` AND the CI quality gate (`overall_status == critical`, because entsoe missing = critical severity).
**Fix**: Distinguish "upstream published no data" (source healthy) from "collector/API failed". New `UpstreamNoDataError(NonRetryableError)` + `CollectorStatus.NO_DATA` + `last_run_no_upstream_data` flag in `base.py`; `collect()` fast-fails without tripping the circuit breaker. `entsoe.py` translates `NoMatchingDataError`→`UpstreamNoDataError`. `validate_pipeline(upstream_empty=…)` downgrades a missing critical feed `critical`→`warning` when it's an upstream gap (a *genuinely* failed critical feed still → critical). `data_fetcher.py` uses separate NL/DE collector instances (no flag race across the concurrent gather), skips retry rounds for upstream-empty feeds, and splits the final validation so an upstream gap logs a warning and keeps publishing while a real failure still exits 1. **Lesson**: "Source is up but has no data for this window" and "the collector/API broke" look identical when both collapse to `dataset is None` — and treating them the same lets one critical feed's *upstream* outage take down 20 healthy feeds. Give the benign case its own exception/status so the pipeline can degrade instead of fail. Also: before writing a fix, reproduce against the live API — the first hypothesis here (a pagination-terminal-ack bug in entsoe-py) was *wrong*; probing proved offset=0 itself had no data and the library was behaving correctly. And a re-run of a data-collection failure only helps if the failure was transient — confirm transient-vs-structural before spending 16 min re-running.
**Follow-up (multi-model review battery, PR #39 post-merge)**: three models (Opus + Sonnet correctness, security lens) reviewed the merged diff. No Critical; core mechanism sound. Actioned in a follow-up: (1) *graceful-degrade must not be silent forever* — added consecutive-upstream-empty escalation (`UPSTREAM_EMPTY_ESCALATION_RUNS`=3, streak in committed `data/_upstream_empty_streak.json`) that re-escalates a sustained gap to a hard failure; (2) demoted the `_retry_with_backoff` `UpstreamNoDataError` log ERROR→WARNING (it was a false alarm); (3) `entsoe._parse_response` now *raises* when the API returned rows but none fall in-window after parsing (a tz/window bug) instead of silently publishing an empty series — a distinct masking path from the upstream gap. **Rejected** a reviewer suggestion to "unify" `CRITICAL_DATASETS` (run-fail/retry set = {entsoe, entsoe_de}) with the quality registry's `'critical'` set ({entsoe, energy_zero}) — they're *intentionally* different (DE day-ahead hard-fails the run but is a coupled/secondary signal that shouldn't block the CI publish gate; a "stable contract" test locks the registry side). Documented the distinction at `CRITICAL_DATASETS` rather than changing it. Lesson-of-the-lesson: a review battery flags intentional-but-undocumented asymmetries as bugs — the fix is often documentation + a locking test, not a code change; verify against existing tests before "fixing" what they deliberately encode.
