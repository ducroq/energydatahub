### ENTSO-E API 503 caused silent data degradation (2026-03-27) [RESOLVED]
**Problem**: ENTSO-E API returned 503 on 2026-03-26. Price files lost `entsoe` and `entsoe_de` keys (27KB → 7.8KB). Augur's consumer price forecast broke completely. Scan revealed 123 degraded files (43% of all price data) going back to Sep 2025.
**Root cause**: `CombinedDataSet.add_dataset()` silently returned on `None` datasets. `data_fetcher.py` saved degraded files without warning. Workflow reported success.
**Fix**: (1) `add_dataset()` now logs WARNING on None datasets. (2) Critical collectors (`entsoe`, `entsoe_de`) retry up to 3 rounds × 5 min delay. (3) Workflow exits non-zero if critical data still missing after retries. (4) Backfilled 100 files with `scripts/backfill_entsoe.py` (2026-03-28).
**Negative result**: The BaseCollector's built-in retry (3 attempts, 1-60s backoff) is too fast for API-wide outages — the API was down for the entire collection window. Longer delays between retry rounds were needed.
**Remaining**: 26 early files (Sep-Oct 2025) have malformed timestamps in non-ENTSO-E datasets, preventing re-save.

### Encrypted data files can't be read back for accumulation (2026-03-27) [RESOLVED]
**Problem**: Market history accumulation needs to read the previous `market_history.json`, but published files are encrypted.
**Root cause**: `save_data_file()` encrypts when `encryption=True`. The code used raw `json.load()` on the encrypted file.
**Fix**: (2026-03-30) Replaced raw `json.load()` with `load_data_file(handler=handler)` which auto-detects encrypted vs plain JSON and decrypts as needed. Market history now accumulates correctly across runs.

### ENTSO-E API returns different column formats per country (2026-03-27) [RESOLVED]
**Problem**: `EntsoeGenerationCollector._parse_response()` matches columns by lowercase name (e.g., "nuclear" in column name). Some countries return MultiIndex DataFrames, others return flat columns.
**Root cause**: The `entsoe-py` library normalizes differently depending on what the ENTSO-E API returns per country.
**Fix**: The existing collector handles both Series and DataFrame with `isinstance` checks. When adding generation mix for NL/DE/BE, same logic applies — no code change needed, just awareness. Marked resolved 2026-06-07 (curate): the code is in place and has shipped reliably; the awareness item belongs in code comments, not the gotcha log.

### GIE gas storage timestamps were integer indices (2026-03-31) [RESOLVED]
**Problem**: 75 of 90 historical gas_storage files had integer keys ('0', '1', '2') instead of ISO timestamps. Data values were intact but unusable without proper time alignment.
**Root cause**: `pd.concat(ignore_index=True)` in `_fetch_raw_data` dropped the `gasDayStart` index. `_parse_response` fell back to `str(_)` which gave integer indices.
**Fix**: (1) Collector fix: preserve `gasDayStart` index in concat. (2) Backfill: `scripts/backfill_gas_storage.py` reconstructed timestamps from metadata `start_time` + daily offset. 75 files patched, 0 errors. Script is idempotent.

### Per-item ENTSO-E retries were missing (2026-03-30) [RESOLVED]
**Problem**: ENTSO-E 503 errors on individual borders/countries were not retried. BaseCollector's `_retry_with_backoff` wraps the entire `_fetch_raw_data`, but inside the loop, individual failures were caught and swallowed.
**Root cause**: Collectors loop over borders/countries and catch exceptions per-item to continue with others, bypassing the outer retry mechanism.
**Fix**: Added `BaseCollector._retry_single()` — retries individual sub-requests (3 attempts, 2s initial backoff). Applied to entsoe_flows, entsoe_wind, entsoe_load, entsoe_generation. Result: cross-border flows went from 4/10 borders to 10/10 on a run with ENTSO-E instability.

### Unicode arrow broke Windows console logging (2026-03-30) [RESOLVED]
**Problem**: `entsoe_flows.py` uses `→` (U+2192) in border names (data keys). Windows cp1252 console can't encode it, causing every flow log line to throw `UnicodeEncodeError`. Data collection still worked, but logs were unreadable.
**Fix**: Configured logging StreamHandler with UTF-8 encoding in `data_fetcher.py`. Border names kept as `→` to preserve backward compatibility with historical data files.
**Negative result**: Initially replaced `→` with `->` in data keys, which would have broken schema compatibility with 145+ historical cross-border flow files. Reverted.

### Optional result unpacking uses fragile index counting (2026-03-27)
**Problem**: Adding a new fixed task to `asyncio.gather()` requires updating the slice index (e.g., `results[:14]` -> `results[:15]`) and `optional_idx`. Easy to miscount.
**Root cause**: Results are unpacked positionally from a flat list. Optional collectors are appended conditionally, making the index depend on which optionals are enabled.
**Fix**: Bumped index from 14 to 15 when adding generation mix. Future improvement: consider using a dict-based result collection pattern instead of positional unpacking.

### Coverage gate failed spuriously on filtered test job after PR #16 merge (2026-06-07) [RESOLVED]
**Problem**: After PR #16 merged, the Tests workflow's "Run critical timezone tests only" job failed with `Required test coverage of 20% not reached. Total coverage: 19.50%`. Generated an email notification even though HEAD of main was green (subsequent PR #18 + PR #20 merges added covered code and pushed coverage back over threshold).

### Shared OpenMeteo Semaphore(5) caused buurt collector timeouts (2026-06-07) [RESOLVED]
**Problem**: After PR #18 merged (shared `Semaphore(5)` across 6 OpenMeteo collectors), two consecutive workflow_dispatch runs (27086755763, 27087174593) failed identically: late-scheduled collectors (offshore wind + both buurts) consistently got `Connection timeout to host api.open-meteo.com` while early strategic/demand collectors succeeded. The data-quality gate correctly blocked publish, but air_quality_buurt envelope verification was blocked.
**Root cause**: Open-Meteo's CDN applies a per-source connection-cooldown window after a request burst. With cap=5 < 6 collectors, 28 location requests queued FIFO behind the shared semaphore; the last ~7 (offshore + buurt) arrived during the cooldown. Pre-#11 the 6 collectors had their own `Semaphore(1)` each — 6 parallel sessions instead of one serialised queue — which avoided the trigger.
**Fix**: PR #23 — bumped `OPENMETEO_SEMAPHORE_CAP` from 5 → 6 (== collector count), restoring the effective pre-#11 concurrency pattern. Also added `fetch_location_with_retry` with exponential backoff (1s → 2s → 4s, max 3 attempts) per location for belt-and-suspenders resilience against any future transient.
**Lesson**: When sharing a budget across N independent consumers, FIFO < N means the late arrivals can hit upstream throttling that the early ones triggered. Cap should equal consumer count unless the consumers' arrival pattern is interleaved by design.

### Worktree path mismatch broke 4 of 12 reviewer agents (2026-06-07)
**Problem**: Created git worktrees at `/tmp/wt-prN` via bash for the multi-model review battery on 4 PRs. 4 of 12 spawned reviewer agents (Opus code-reviewer for PRs #16/#18/#19 + Sonnet for #16) returned "I cannot review code I cannot read" — their Read tool couldn't access the paths.
**Root cause**: On Windows, bash's `/tmp` maps to `C:\Users\<user>\AppData\Local\Temp` (a separate per-user temp dir, not the system `/tmp` that POSIX expects). The reviewer agents tried `/tmp/wt-prN` and got "not found"; only the agents that happened to try the Windows path succeeded.
**Fix (one-shot)**: Per-agent path translation worked for some agents; others gave up. For the failed ones, no review was produced — partial battery coverage.
**Lesson**: When briefing sub-agents that will use Read/Glob, give them OS-native absolute paths. On Windows, prefer `%LOCALAPPDATA%\Temp\...` or pass the diff inline rather than relying on `/tmp`. Better: have the orchestrator generate diffs and embed them in the agent prompts directly.
**Positive side-effect**: Three of the four agents that couldn't read the code refused to fabricate findings (cited the "When something is unclear, ask rather than guess" harness-defense rule), which is the desired behavior. The fix is upstream — don't put them in that position.

### Collector's `_fetch_raw_data` raised ValueError when it meant "permanent failure" (2026-06-07) [RESOLVED]
**Problem**: `EntsoeHydroCollector._fetch_raw_data` raised `ValueError("No reservoir data returned")` when all requested zones returned empty after their per-zone retries. Code-reviewer agent flagged this as a BLOCKER: `BaseCollector._retry_with_backoff` catches `except Exception` and triggers up to `max_attempts` more outer retries — each re-running all zones, all returning empty, all raising ValueError again. The bail-out intent was completely defeated.
**Root cause**: There are two distinct exception contracts in `BaseCollector` and they look identical from the inside of a collector:
  - Anything that inherits from `Exception` → caught by the outer retry loop, retried with backoff.
  - `NonRetryableError` (issue #25) → re-raised immediately, no further attempts.

  When a collector decides "I've tried everything I can; the upstream is permanently empty for this window", it must raise `NonRetryableError`, not a plain `ValueError`/`RuntimeError`. The exception name is the signal, not the message.
**Fix**: Changed the raise site to `raise NonRetryableError("No reservoir data returned...")`. Added a regression test that mocks all zones to return None and asserts exactly `num_zones × 1 outer attempt = num_zones` `_retry_single` calls (not `num_zones × max_attempts`). The test would have failed loudly under the buggy ValueError path.
**Lesson**: Any "we tried, it's not coming back" exit from `_fetch_raw_data` is a `NonRetryableError`, never a plain Exception. If a future collector adopts the `_http_classifier.raise_if_permanent` helper, that's already correct; the trap is the *manual* raise sites where it's tempting to reach for `ValueError`. Code comment in `collectors/entsoe_hydro.py` now documents this.

### Three parallel registries with the same key shape — collapse before adding the third (2026-06-07) [RESOLVED]
**Problem**: `utils/data_quality.py` grew three lists for missing-dataset severity: `REQUIRED_DATASETS` (critical), `EXPECTED_DATASETS` (info-only logging), `WARNING_IF_MISSING_DATASETS` (warning, added today for the #25 soft-gate). Each was a `List[str]` keyed on dataset name. The `PipelineQualityReport.status` property and `validate_pipeline` loop consulted all three with implicit precedence. `EXPECTED_DATASETS` was effectively dead — it duplicated the critical entries plus a few info-flake ones, and didn't influence status at all. The refactoring-guide reviewer caught this explicitly: "do this now, before a 4th list".
**Root cause**: Each registry was added in a different commit for a different bug (a74f662, 7673750, etc.). Nobody made the structural call to consolidate because each individual list looked reasonable. The shape "three lists keyed on the same identifier, with implicit severity ranking encoded in which list a name appears in" was only visible from above.
**Fix**: Replaced with a single `DATASET_MISSING_SEVERITY: Dict[str, str]` where values are `'critical' / 'warning' / 'info'`. `validate_pipeline` reduces to one pass; the status ladder consults a single registry. The three named lists are preserved as back-compat shims derived from the dict so any importer keeps working.
**Lesson**: When you reach for a *third* parallel registry with the same key shape (the same identifier in each, the same lookup site iterating all of them), consolidate then. Two parallel lists is the limit before the precedence becomes implicit and ambiguous. The compile-time signal is "I'm about to add another list" — if you're naming it `*_IF_MISSING_*`, that's already a hint that severity is the real dimension and you want a dict keyed by name with severity as value.

### `validate_value_ranges` silently skipped 2-level-nested feeds for 3+ months (2026-06-07) [RESOLVED]
**Problem**: While shipping #3, the security-auditor flagged that the new `hydro_reservoir` data_type would have no per-field range validation. The fix was supposed to be a one-line `FIELD_RANGES_BY_TYPE['hydro_reservoir']` entry mirroring `gas_storage`. After adding it, a poison test (`reservoir_mwh=9.9e9`) DIDN'T fire the range check. Investigation: weather poisoning (`temperature=9999`) also didn't fire. The validator had been silently no-op'ing range checks on every 2-level-nested feed — `{location: {ts: {field: value}}}` — since the `weather` per-field path was introduced.
**Root cause**: `validate_value_ranges` iterated `data.items()` once and treated each `value` as a `{field: scalar}` dict. For 2-level structures the `value` was `{ts: {fields}}` — the inner loop saw `sub_value` as a dict, hit the `if not isinstance(sub_value, (int, float)): continue` branch, and skipped. No error, no warning, just silent zero-issue reports.
**Fix**: Added `_flatten_to_timestamp_records` helper at the function entry: if the top-level keys aren't ISO timestamps (heuristic — try `datetime.fromisoformat` on the first few), descend one level. Restored validation across `weather_forecast_multi_location`, `weather_forecast_buurt`, `demand_weather_forecast`, `nordic_hydro`. The fix surfaced *real* range-bound issues on `solar_forecast_buurt`, `solar_forecast`, and `load_forecast` that had been masked the whole time (filed as #28).
**Lesson**: Silent-pass quality gates rot. When adding a new entry to a validation registry, run a poison test to confirm it actually fires — don't just trust that registration = enforcement. Bonus: this is the SECOND silent-validation incident this month (the first was the GoogleWeather `API_KEY_INVALID` returning success exit codes for 7 months). The pattern is: "validator returns empty issues list" looks identical to "data is clean" — make sure the path is exercised.

### Schema-drift tripwire conflated catalog drift with shape drift (2026-06-08) [RESOLVED]
**Problem**: While verifying the 2026-06-07 daily run as part of the next-session pickup, the 17:07 schedule run's schema-drift step reported `feeds added: ['air_quality_buurt.json']` with `::warning::Schema drift detected ... but schema_version did not change`. Confirmed by diffing the two sidecar commits (`10fe22e` baseline vs `8590b64`): the luchtmeetnet collector failed transiently in the 16:58 dispatch baseline run and recovered by the 17:07 schedule run. The "added" entry was a transient collector recovery, not a schema event. Under the current `--warn-only` mode this was harmless noise, but the 2026-06-21 plan to flip to fail-mode would have errored every CI run that contained even one transiently-flaky collector.
**Root cause**: `scripts/detect_schema_drift.py` treated `feeds_changed` (within-feed shape diff), `feeds_added`, and `feeds_removed` as the same severity class — any of them without a `CURRENT_SCHEMA_VERSION` bump → exit 1. But `schema_version` versions the envelope/migration shape of existing feeds, NOT the catalog of which feeds the pipeline currently publishes. A collector flaking off and back on is operational; a collector being retired is operational; only a within-feed shape change is the silent-shape-break failure mode the tripwire exists to catch.
**Fix**: Split the fail-mode classification in `detect_schema_drift.py:main`. `feeds_changed` without version bump → exit 1 (the real target). `feeds_added` / `feeds_removed` without `feeds_changed` → emit `::warning::Catalog drift: ...` and exit 0, even in fail-mode. Mixed case (within-feed shape change + catalog drift) → fail (shape change dominates). Added three tests: `test_feeds_added_only_exits_0_with_warning`, `test_feeds_removed_only_exits_0_with_warning`, `test_within_feed_change_plus_catalog_drift_still_fails`. The original `test_drift_without_version_bump_exits_1` continues to pass because it uses `feeds_changed`. Module docstring updated to call out "Catalog vs shape drift" explicitly.
**Lesson**: When a tripwire fires three different events through the same exit-code path, ask whether the user (the future operator reading the CI alert) needs the same response for all three. Here the answer was no: shape drift is "did someone forget to bump the version?" while catalog drift is "did a collector come back / get retired?" — two completely different operational questions. The fix preserves the original silent-shape-break detection (which was the tripwire's whole purpose per #27) while removing the false-positive class that would have blocked the 2026-06-21 fail-mode flip. Also a reminder: when scheduling a "flip to fail-mode" date based on N weeks of clean warn-only runs, audit the actual warnings during the bedding-in period — don't just count green runs. The 17:07 warning was on the record; only the cross-session pickup spotted it.

### TenneT `balance-delta` endpoint retired without library update — secondary signal failure killed primary signal (2026-06-08) [RESOLVED]
**Problem**: Issue #25's original diagnosis (422→422→429 cascade on `balance-delta-high-res`) was correct when filed but the API has since changed: TenneT retired the `balance-delta-high-res` endpoint, renamed to `balance-delta`, and the new endpoint now returns HTTP 404 for every window tested (yesterday → 30 days back via `scripts/probe_tennet_windows.py`). `settlement_prices` continues to work normally (96 rows/day = full 15-min resolution). Because `_fetch_raw_data` ran both calls inside one try-block, the dead `balance_delta` endpoint's 404 propagated up and the classifier from 7673750 (correctly) bailed out the entire fetch — taking the working `settlement_prices` payload down with it. Net effect: `grid_imbalance` silently absent from the published feed for weeks, and the only signal of degradation was the soft-gate's `warning` status.
**Root cause**: Two distinct issues:
  1. **Library lag**: `tenneteu-py` still calls `/publications/v1/balance-delta` but TenneT's actual endpoint either renamed again or is genuinely retired. The library hasn't been updated.
  2. **Coupled error path**: A secondary signal's permanent error killed the primary signal because they shared a try-block. The classifier did its job; the structure around it forced an all-or-nothing outcome.
**Fix**: Split `_fetch_raw_data` so each API call has its own try-block. `settlement_prices` keeps the existing classifier behaviour (permanent → `NonRetryableError`, transient → retry). `balance_delta` permanent errors (404/422/400/401/403) are caught and substituted with `None`; transient errors still propagate so the retry loop has a chance. The downstream `_parse_response` already defaults `balance_delta=0.0` / `direction='balanced'` when `balance_delta_df` is None — so degraded but usable `grid_imbalance` records still publish, restoring the feed. Added 4 tests covering the split + a regression for the original classifier on settlement_prices. Auto-heals if/when TenneT restores the endpoint.
**Lesson**: When a try-block wraps multiple API calls whose failures have different operational meanings, split it. The classifier-bail-out pattern (`NonRetryableError`) is correct for "no point retrying this", but conflating "primary signal failed" with "secondary signal failed" forces both to take the same action — and the primary's action is "give up entirely" which silently retires the whole feed. The second insight: when an issue's diagnosis ages, re-probe the upstream before re-acting on the original analysis. The 2026-06-08 root-cause work changed the recommended fix from "investigate window boundary" to "split the try-block" because the API shape changed under us in the intervening weeks.

### Shallow `dict(issue)` mutated source collector metadata via nested `details` (2026-06-07) [RESOLVED]
**Problem**: PR #20's `assemble_buurt_air_envelope` did `tagged = dict(issue); tagged.setdefault('details', {})['location'] = loc['name']` to add a per-buurt location tag to aggregated quality issues. Two independent reviewers (Opus and Sonnet, separate models) flagged that the shallow copy shared the inner `details` dict with the source — adding `location` mutated the collector's own metadata.
**Root cause**: `dict(other_dict)` is a shallow copy. The top-level dict is new, but any nested mutable values (lists, dicts) are still shared references.
**Fix**: Use `copy.deepcopy(issue)` (or rebuild the dict explicitly with new inner dicts). Added a regression test `test_issue_details_location_tag_does_not_mutate_source` that builds an issue, runs the assembler, and asserts the original `details` dict doesn't gain a `location` key.
**Lesson**: When mutating a "copied" dict that came from a caller's state, default to `copy.deepcopy` unless you've audited every nested value. Multi-model review caught this where a single reviewer might have rationalised it as fine. Worth keeping the multi-model pattern for production data-flow code.
